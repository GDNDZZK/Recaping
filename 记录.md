## 项目简介

Recaping 是一款专注于多媒体录音记录的应用程序。它允许用户在录音过程中同时记录照片、视频片段和文字笔记，形成完整的多媒体时间轴记录。录音完成后，用户可以回放录音并查看所有关联的多媒体内容。

## 核心功能

1. **录音功能**：持续录音，自动分段存储（15秒/段）
2. **多媒体记录**：在录音过程中拍摄照片、录制短视频片段（5秒/段）、添加文字笔记
3. **时间轴展示**：双时间轴模型（录音时间轴 + 多媒体时间轴）
4. **回放功能**：回放录音，同步显示多媒体内容
5. **AI 功能**：录音转文字、智能摘要、关键词提取

## 开发规则与注意事项

### 代码规范

1. **命名规范**
   - 文件命名：小写 + 下划线（snake_case）
   - 类命名：大驼峰（PascalCase）
   - 变量/函数：小驼峰（camelCase）
   - 常量：全大写 + 下划线（SCREAMING_SNAKE_CASE）

2. **代码组织**
   - 按功能模块组织代码
   - 每个文件只包含一个主要类
   - 避免过长的函数，保持单一职责

3. **注释规范**
   - 公共 API 必须添加文档注释（///）
   - 复杂逻辑添加行内注释
   - TODO 标记使用 `// TODO(GDNDZZK): 描述`

### 数据库规范

1. **表设计**
   - 所有表必须包含 `id`（主键）、`created_at`、`updated_at` 字段
   - 外键使用 `_id` 后缀
   - 表名使用复数形式

2. **数据类型**
   - 时间戳使用 INTEGER（Unix 时间戳，毫秒）
   - 布尔值使用 INTEGER（0/1）
   - 大文本使用 TEXT

### 文件管理

1. **存储路径**
   - 应用数据：应用私有目录
   - 录音文件：`/audio/`
   - 照片文件：`/photos/`
   - 视频文件：`/videos/`
   - 导出文件：`/exports/`

2. **文件命名**
   - 使用时间戳 + 随机字符串
   - 格式：`{timestamp}_{random}.{ext}`

### 性能注意事项

1. **录音时**
   - 避免在主线程执行耗时操作
   - 音频写入使用异步操作
   - 定期清理内存缓存

2. **数据库操作**
   - 批量操作使用事务
   - 查询使用索引字段
   - 大数据量分页加载

3. **UI 渲染**
   - 列表使用懒加载
   - 图片使用缓存
   - 避免不必要的重建

### 安全注意事项

1. **权限管理**
   - 录音权限必须动态申请
   - 相机权限必须动态申请
   - 存储权限适配 Android 10+ Scoped Storage

2. **数据安全**
   - 敏感数据不存储在 SharedPreferences
   - 数据库文件加密（可选）
   - 导出文件注意权限控制

### 版本控制

1. **Git 提交规范**
   - 提交信息使用中文描述
   - 格式：`[类型] 描述`
   - 类型：feat, fix, docs, style, refactor, test, chore

2. **分支管理**
   - main：稳定版本
   - develop：开发分支
   - feature/*：功能分支

### 依赖管理

1. **pubspec.yaml**
   - 精确版本号，避免使用 `any`
   - 定期更新依赖
   - 移除未使用的依赖

2. **常用依赖**

### 2026-04-09（存储方案重构 — 文件系统存储 + 单文件导出导入）

#### 背景
原方案将所有数据（包括照片、视频、音频）作为 BLOB 存入 SQLite，导致：
1. 大照片（未压缩）触发 SQLite "Row too big" 错误
2. 数据库文件过大，性能下降

#### 新方案
- **运行时存储**：每个会话一个目录（`sessions/{session_id}/`），包含 `session.db`（SQLite 数据库，只存元数据和文件路径引用）和 `audio/`、`photos/`、`videos/`、`thumbnails/` 子目录
- **导出格式**：ZIP 文件（.recp），包含整个会话目录
- **导入**：解压 ZIP 文件到 sessions 目录

#### 修改的文件
- **数据模型**：
  - `audio_chunk.dart`：`Uint8List data` → `String filePath`
  - `photo.dart`：`Uint8List data` + `Uint8List thumbnail` → `String filePath` + `String thumbnailPath`
  - `video_chunk.dart`：`Uint8List data` + `Uint8List? thumbnail` → `String filePath` + `String? thumbnailPath`
  - `timeline_event.dart`：`Uint8List? thumbnail` → `String? thumbnailPath`
- **数据库层**：
  - `database_helper.dart`：新增 `sessionDirPath()`、`ensureSessionDirs()`、`deleteSessionDirectory()` 方法；数据库文件从 `.recp` 改为会话目录内的 `session.db`
  - `session_database.dart`：建表 SQL 中 BLOB 字段改为 TEXT（文件路径）；新增 `resolvePath()` 方法；时间轴事件查询返回绝对路径
- **服务层**：
  - `storage_service.dart`：`createSession()` 创建目录结构；`deleteSession()` 删除整个目录；`getDetailedStorageStats()` 遍历目录统计大小；`compressSessionMedia()` 从文件读取原图
  - `recording_service.dart`：录音分片保存到 `audio/chunk_{index}.aac` 文件，数据库存路径引用
  - `timeline_service.dart`：照片写入 `photos/{id}.jpeg`，缩略图写入 `thumbnails/{id}_thumb.jpeg`，视频写入 `videos/{video_id}.mp4`
  - `audio_playback_service.dart`：从文件系统读取音频分片合并播放
  - `export_service.dart`：使用 `archive` 包将整个会话目录打包为 ZIP（.recp），导入时解压 ZIP
- **UI 组件**：
  - `timeline_event_item.dart`：缩略图从 `Image.memory()` 改为 `Image.file()`
  - `image_viewer.dart`：支持从文件路径加载图片
  - `playback_page.dart`：照片查看使用文件路径
  - `ai_page.dart`：音频数据从文件系统读取合并
- **依赖**：
  - `pubspec.yaml`：添加 `archive: ^3.4.10` 依赖

- flutter analyze 通过，0 issues

#### 调试背景
之前的两次修复（`ref.listen` → `ref.listenManual`、`addPostFrameCallback` + `setState` 移到 `.then()`）均未能彻底解决录音中点击名称重命名时的崩溃。

#### 日志分析关键发现

**1. 错误发生在对话框打开时，而非关闭时**

时间线：
| totalMs | 事件 |
|---------|------|
| ~1401ms | 用户点击标题 → `_showEditTitleDialog` 开始 |
| ~1502ms | `addPostFrameCallback` 执行 → `showDialog` builder 调用 |

这说明之前的修复假设错误发生在 dialog 关闭时是错误的。

**2. 错误链**

1. `showDialog` 在 `addPostFrameCallback` 中 push 新路由 → 触发 `RecordPage.build()` 重建
2. `build()` 中 7 个 `ref.watch()` 重新向 `_InheritedProviderScopeElement` 注册依赖
3. Dialog 路由的 widget tree 初始化/过渡过程中，某些 InheritedWidget element 被 deactivate
4. 这些 element 上有来自步骤 2 注册的 dependents
5. 触发 `_dependents.isEmpty` 断言

**3. 高频重建是根本原因**

录音期间 `build()` 每 ~100ms 被调用一次（来自 `totalElapsedMsProvider` 等 StreamProvider 的更新）。当 `showDialog` push 新路由时，这些高频更新与 dialog 的 widget tree 过渡产生竞态，导致 InheritedWidget 依赖追踪进入不一致状态。

**4. TextEditingController 提前 dispose 的谜题**

日志显示 `editController` 在 dialog 仍然显示时就被 dispose（`TextEditingController was used after being disposed`），但代码中 `editController.dispose()` 仅在 `.then()` 回调中调用。可能原因是 `showDialog` 返回的 Future 在 widget tree 完全稳定之前就完成了，或者 Builder 被多次调用导致 controller 重新绑定。

#### 可能的根本原因（按可能性排序）

1. **`showDialog` push 路由与高频 StreamProvider 重建的竞态条件**（最可能）
   - `build()` 在高频 StreamProvider 驱动下不断重建
   - `showDialog` push 路由触发 widget tree 过渡
   - 两者在同一帧内竞争 InheritedWidget 依赖注册

2. **`addPostFrameCallback` 内 `showDialog` 的时序问题**
   - `addPostFrameCallback` 在帧结束时执行
   - 但 StreamProvider 可能在微任务队列中排队了更新
   - `showDialog` 的 route 动画与 Provider 更新交叉

3. **`RecordingTimeline` 子组件在 dialog 过渡期间的状态变化**
   - `RecordingTimeline._scrollToBottom()` 也使用 `addPostFrameCallback`
   - 可能与 dialog 的 route 过渡冲突

#### 建议修复方向

1. **抑制 `build()` 在 dialog 显示期间的频率**：在 `build()` 开头检查是否有 dialog 正在显示，跳过不必要的 `ref.watch` 调用
2. **使用 `Overlay` 而非 `showDialog`**：`showDialog` 会 push 新路由，改用 `Overlay.insert()` 在当前路由上叠加
3. **将 `showDialog` 移到微任务队列**：使用 `Future.microtask()` 进一步延迟 `showDialog` 的执行时机
