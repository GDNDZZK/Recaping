- 改动代码后需要运行flutter analyze确保通过
- Flutter 引擎会自动将 Android intent data URI 作为深度链接路由到 go_router，需要在 go_router 中添加 errorBuilder 处理无法匹配的路由（如 content:// URI），否则会显示"页面未找到"错误
- receive_sharing_intent 插件已内置 content URI 处理（FileDirectory.getAbsolutePath 会将 content 复制到缓存目录并返回本地路径），不需要额外处理
- ConsumerStatefulWidget 需要在其**上方**存在 ProviderScope 才能使用 ref。ProviderScope 不能放在 ConsumerStatefulWidget 的 build 方法内部，必须放在更上层（通常是 main.dart 的 runApp 中）
- `FilePicker.saveFile()` 在 Android 上返回的是 content URI 路径（如 `/document/primary:Download/file.recp`），不是文件系统路径。`File` 操作对 content URI 路径无效（`exists()` 返回 false）。需要将 content URI 路径转换为文件系统路径（如 `/storage/emulated/0/Download/file.recp`）才能进行文件操作
- `file_picker` 的 Android 实现使用 Apache Tika 检测 MIME 类型，ZIP 格式的自定义扩展名文件（如 `.recp`）会被检测为 `application/zip`，导致 SAF 自动追加 `.zip` 后缀。无法从 Dart 侧覆盖此行为。解决方案：使用 `FilePicker.getDirectoryPath()` 让用户选择目录，手动拼接文件名并 `File.copy()`，完全绕过 SAF 的 MIME 检测
- `file_picker` v11 API 变更：`FilePicker.platform.getDirectoryPath()` → `FilePicker.getDirectoryPath()`（静态方法直接调用，不再通过 `.platform`）
- **go_router 版本兼容性**：go_router 13+ 版本与 Flutter 的 PopScope 组件存在兼容性问题，导致系统返回键无法被正确拦截。降级到 12.x 版本并使用 WillPopScope 是稳定的解决方案
- **Android 13+ 返回键机制**：Android 13 引入了 `OnBackInvokedCallback` 机制，需要在 `AndroidManifest.xml` 的 `<activity>` 标签中设置 `android:enableOnBackInvokedCallback="true"`，否则系统返回键无法被正确拦截（日志会提示 `OnBackInvokedCallback is not enabled for the application`）
- **WillPopScope vs PopScope**：go_router 12.x 版本对 WillPopScope 的支持更稳定。`WillPopScope.onWillPop` 回调返回 `Future<bool>`，返回 `false` 阻止返回，返回 `true` 允许返回。而 go_router 13+ 与 PopScope 存在冲突
- **必须重新安装应用**：修改 `AndroidManifest.xml` 后，必须卸载旧应用后重新安装，仅热重启无效（因为 AndroidManifest.xml 是原生配置，需要重新安装才能生效）
