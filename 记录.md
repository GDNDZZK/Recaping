## 项目简介

Recaping 是一款专注于多媒体录音记录的应用程序。它允许用户在录音过程中同时记录照片、视频片段和文字笔记，形成完整的多媒体时间轴记录。录音完成后，用户可以回放录音并查看所有关联的多媒体内容。

## 核心功能

1. **录音功能**：持续录音，自动分段存储（15秒/段）
2. **多媒体记录**：在录音过程中拍摄照片、录制短视频片段（5秒/段）、添加文字笔记
3. **时间轴展示**：双时间轴模型（录音时间轴 + 多媒体时间轴）
4. **回放功能**：回放录音，同步显示多媒体内容
5. **AI 功能**：录音转文字、智能摘要、关键词提取

## 开发规则与注意事项

### 代码规范

1. **命名规范**
   - 文件命名：小写 + 下划线（snake_case）
   - 类命名：大驼峰（PascalCase）
   - 变量/函数：小驼峰（camelCase）
   - 常量：全大写 + 下划线（SCREAMING_SNAKE_CASE）

2. **代码组织**
   - 按功能模块组织代码
   - 每个文件只包含一个主要类
   - 避免过长的函数，保持单一职责

3. **注释规范**
   - 公共 API 必须添加文档注释（///）
   - 复杂逻辑添加行内注释
   - TODO 标记使用 `// TODO(GDNDZZK): 描述`

### 数据库规范

1. **表设计**
   - 所有表必须包含 `id`（主键）、`created_at`、`updated_at` 字段
   - 外键使用 `_id` 后缀
   - 表名使用复数形式

2. **数据类型**
   - 时间戳使用 INTEGER（Unix 时间戳，毫秒）
   - 布尔值使用 INTEGER（0/1）
   - 大文本使用 TEXT

### 文件管理

1. **存储路径**
   - 应用数据：应用私有目录
   - 录音文件：`/audio/`
   - 照片文件：`/photos/`
   - 视频文件：`/videos/`
   - 导出文件：`/exports/`

2. **文件命名**
   - 使用时间戳 + 随机字符串
   - 格式：`{timestamp}_{random}.{ext}`

### 性能注意事项

1. **录音时**
   - 避免在主线程执行耗时操作
   - 音频写入使用异步操作
   - 定期清理内存缓存

2. **数据库操作**
   - 批量操作使用事务
   - 查询使用索引字段
   - 大数据量分页加载

3. **UI 渲染**
   - 列表使用懒加载
   - 图片使用缓存
   - 避免不必要的重建

### 安全注意事项

1. **权限管理**
   - 录音权限必须动态申请
   - 相机权限必须动态申请
   - 存储权限适配 Android 10+ Scoped Storage

2. **数据安全**
   - 敏感数据不存储在 SharedPreferences
   - 数据库文件加密（可选）
   - 导出文件注意权限控制

### 版本控制

1. **Git 提交规范**
   - 提交信息使用中文描述
   - 格式：`[类型] 描述`
   - 类型：feat, fix, docs, style, refactor, test, chore

2. **分支管理**
   - main：稳定版本
   - develop：开发分支
   - feature/*：功能分支

### 依赖管理

1. **pubspec.yaml**
   - 精确版本号，避免使用 `any`
   - 定期更新依赖
   - 移除未使用的依赖

2. **常用依赖**

### 2026-04-09（存储方案重构 — 文件系统存储 + 单文件导出导入）

#### 背景
原方案将所有数据（包括照片、视频、音频）作为 BLOB 存入 SQLite，导致：
1. 大照片（未压缩）触发 SQLite "Row too big" 错误
2. 数据库文件过大，性能下降

#### 新方案
- **运行时存储**：每个会话一个目录（`sessions/{session_id}/`），包含 `session.db`（SQLite 数据库，只存元数据和文件路径引用）和 `audio/`、`photos/`、`videos/`、`thumbnails/` 子目录
- **导出格式**：ZIP 文件（.recp），包含整个会话目录
- **导入**：解压 ZIP 文件到 sessions 目录

#### 修改的文件
- **数据模型**：
  - `audio_chunk.dart`：`Uint8List data` → `String filePath`
  - `photo.dart`：`Uint8List data` + `Uint8List thumbnail` → `String filePath` + `String thumbnailPath`
  - `video_chunk.dart`：`Uint8List data` + `Uint8List? thumbnail` → `String filePath` + `String? thumbnailPath`
  - `timeline_event.dart`：`Uint8List? thumbnail` → `String? thumbnailPath`
- **数据库层**：
  - `database_helper.dart`：新增 `sessionDirPath()`、`ensureSessionDirs()`、`deleteSessionDirectory()` 方法；数据库文件从 `.recp` 改为会话目录内的 `session.db`
  - `session_database.dart`：建表 SQL 中 BLOB 字段改为 TEXT（文件路径）；新增 `resolvePath()` 方法；时间轴事件查询返回绝对路径
- **服务层**：
  - `storage_service.dart`：`createSession()` 创建目录结构；`deleteSession()` 删除整个目录；`getDetailedStorageStats()` 遍历目录统计大小；`compressSessionMedia()` 从文件读取原图
  - `recording_service.dart`：录音分片保存到 `audio/chunk_{index}.aac` 文件，数据库存路径引用
  - `timeline_service.dart`：照片写入 `photos/{id}.jpeg`，缩略图写入 `thumbnails/{id}_thumb.jpeg`，视频写入 `videos/{video_id}.mp4`
  - `audio_playback_service.dart`：从文件系统读取音频分片合并播放
  - `export_service.dart`：使用 `archive` 包将整个会话目录打包为 ZIP（.recp），导入时解压 ZIP
- **UI 组件**：
  - `timeline_event_item.dart`：缩略图从 `Image.memory()` 改为 `Image.file()`
  - `image_viewer.dart`：支持从文件路径加载图片
  - `playback_page.dart`：照片查看使用文件路径
  - `ai_page.dart`：音频数据从文件系统读取合并
- **依赖**：
  - `pubspec.yaml`：添加 `archive: ^3.4.10` 依赖

- flutter analyze 通过，0 issues

### 2026-04-29（修复回放页面播放状态显示延迟的 Bug）

#### 问题描述
在回放页面（`lib/pages/playback/playback_page.dart`）中：
1. 初始状态显示"已暂停"（正确）
2. 点击播放按钮后，音频开始播放，但状态指示器仍显示"已暂停"
3. 再次点击后，状态才变为"播放中"

#### 根本原因
[`AudioPlaybackService.play()`](lib/services/audio_playback_service.dart:121) 中 `_setState(PlaybackState.playing)` 在 `await _player.play()` 之后调用。`just_audio` 的 `_player.play()` 返回的 Future 在音频**播放完毕**后才 complete，因此状态永远不会在播放期间更新为 `playing`。

```dart
// 修复前
Future<void> play() async {
  if (_state == PlaybackState.paused || _state == PlaybackState.idle) {
    await _player.play();        // ← 阻塞直到音频播放结束！
    _setState(PlaybackState.playing);  // ← 播放期间永远不会执行
  }
}
```

#### 修复内容
- **修改 `lib/services/audio_playback_service.dart`**：
  - 将 `_setState(PlaybackState.playing)` 移到 `await _player.play()` **之前**
  - 这样状态会立即更新，UI 能即时响应播放状态变化
  - 当 `_player.play()` 完成（音频播完）时，`_playerStateSubscription` 监听器会正确将状态设为 `idle`

```dart
// 修复后
Future<void> play() async {
  if (_state == PlaybackState.paused || _state == PlaybackState.idle) {
    _setState(PlaybackState.playing);  // ← 立即更新状态
    await _player.play();
  }
}
```

### 2026-04-29（重写 AudioPlaybackService 实现总时间轴连续播放）

#### 需求背景
当前 `AudioPlaybackService` 将所有音频 chunk 的原始字节拼接成一个文件播放，存在两个问题：
1. AAC 字节拼接不产生有效音频文件，导致只能播放第一个 chunk
2. 没有总时间轴概念，无法处理录音段之间的静音间隔

#### 修改方案：定时器驱动的总时间轴播放

核心思路：
- 使用 Stopwatch + Timer 驱动总时间轴前进（每 50ms 更新一次）
- 根据当前总时间轴位置，判断落在哪个录音 chunk（或静音区间）
- 进入录音 chunk 时，加载并播放对应的单个 chunk 文件
- 进入静音区间时，暂停音频但继续推进时间轴

### 2026-05-06（修复离开回放页面后音频继续播放的竞态条件）

#### 问题描述
用户反馈：在回放时返回（离开回放页面），音频还在继续播放。再次进入时播放按钮显示"暂停"但实际还在播放，状态没有正确销毁。

#### 根本原因分析

**竞态条件**：`_loadAndPlayChunk()` 中的异步操作在 `stop()` 后仍能恢复执行并触发 `_player.play()`。

具体流程：
1. `_onTick()` → `_handleTimelinePosition()` → `_transitionToChunk()` → `_loadAndPlayChunk()` 开始执行
2. `_loadAndPlayChunk()` 在某个 `await` 点（如 `await _player.setFilePath()`）被挂起
3. 用户点击返回 → `PlaybackPage.dispose()` → `service.stop()`（fire-and-forget，不 await）
4. `stop()` 内部取消 Timer、重置 Stopwatch、调用 `await _player.stop()`
5. 但步骤 2 中挂起的 `_loadAndPlayChunk()` 恢复执行，继续走到 `_player.play()`
6. 音频重新开始播放，页面已销毁，无法控制

**关键问题**：
- `stop()` 是 async 方法，但 `dispose()` 不是 async 的，无法 await
- `_player.play()` 是 fire-and-forget（不 await），可能在 `stop()` 完成后才开始执行
- `_isTransitioning` 标志在 `stop()` 中被重置，但 in-flight 的 `_loadAndPlayChunk()` 仍持有旧的执行上下文

#### 修复内容

**修改 `lib/services/audio_playback_service.dart`：**

#### 根本原因分析

**竞态条件**：[`_loadAndPlayChunk()`](lib/services/audio_playback_service.dart:479) 中的 `_player.play()` 是 fire-and-forget 调用（不 await），即使 `_isStopped` 检查通过，`_player.play()` 内部的 just_audio/ExoPlayer 异步操作仍会在 `stop()` 后继续执行，创建新的 AudioTrack/MediaCodec 实例并开始播放。

### 2026-05-07（修复返回后音频继续播放 — ref 失效导致 stop() 未调用）

#### 问题描述
用户反复反馈：从回放页面返回后，音频仍在继续播放。之前的多次修复（`_isStopped` 布尔标志、代数计数器 `_generation`）都集中在 `AudioPlaybackService` 服务层，但问题始终未解决。

#### 根本原因

**问题不在服务层，而在页面层。** [`PlaybackPage.dispose()`](lib/pages/playback/playback_page.dart:51) 中通过 `ref.read(playbackServiceProvider)` 获取服务实例，但 Riverpod 的 `ConsumerState` 在 widget 的 Element 被 unmount 后会立即使 `ref` 失效。`dispose()` 执行时 Element 已被 unmount，`ref.read()` 抛出 `Bad state: Cannot use "ref" after the widget was disposed.` 异常。由于 try-catch 静默吞掉了这个异常，`service.stop()` **从未被调用**。

这导致：
- `_generation` 从未递增 → in-flight 异步操作不会失效
- `_timelineTimer` 从未取消 → Timer 继续每 50ms 触发

#### 修复内容

**修改 [`lib/pages/playback/playback_page.dart`](lib/pages/playback/playback_page.dart)：**

1. **新增 `_cachedService` 字段**：
   ```dart
   /// 缓存的播放服务引用。
   /// 在 [build()] 中通过 [ref] 获取并缓存，避免 [dispose()] 时 [ref] 已失效
   /// （Riverpod 在 widget dispose 后不允许使用 ref）导致无法调用 [stop()]。
   AudioPlaybackService? _cachedService;
   ```

2. **在 `build()` 方法中缓存服务引用**：
   ```dart
   // 缓存服务引用：ref 在 build() 中始终有效，但在 dispose() 中已失效。
   // 使用 ??= 确保只赋值一次，后续 build() 调用不会重复赋值。
   _cachedService ??= ref.read(playbackServiceProvider);
   ```

3. **重写 `dispose()` 方法使用缓存引用**：
   ```dart
   @override
   void dispose() {
     debugPrint('[PlaybackPage] dispose() called');
     final service = _cachedService;
     if (service != null) {
       debugPrint('[PlaybackPage] calling service.stop(), current state=${service.state}');
       service.stop();
     } else {
       debugPrint('[PlaybackPage] no cached service available');
     }
     super.dispose();
   }
   ```

#### 为什么这个修复是正确的

1. **`playbackServiceProvider` 是普通 `Provider`**（非 AutoDispose）：服务实例在整个 app 生命周期内保持不变，缓存引用不会失效
2. **`build()` 中 `ref` 始终有效**：Riverpod 保证 `build()` 执行时 Element 处于 mounted 状态
3. **`??=` 确保只赋值一次**：避免每次 `build()` 重复赋值（虽然赋值相同实例也无害）
4. **`dispose()` 中不再依赖 `ref`**：使用缓存的直接引用，绕过 Riverpod 的生命周期限制

#### 技术要点

- **Riverpod `ref` 生命周期**：`ConsumerState` 的 `ref` 在 Element unmount 后立即失效（抛出 `Bad state` 异常），`dispose()` 在 unmount 之后执行
- **`Provider` vs `AutoDisposeProvider`**：普通 `Provider` 创建的实例不会被自动释放，缓存引用安全
- **缓存模式**：在 `build()` 中获取并缓存外部依赖，在 `dispose()` 中使用缓存引用，是 Flutter/Riverpod 中处理 dispose 生命周期问题的标准模式

### 2026-05-07 修复 Android 打开 .recp 文件时 content URI 路由错误

#### 问题描述
从 Android 文件管理器打开 .recp 文件时，应用跳转到"页面未找到"错误界面，错误提示：`GoException: no routes for location: content://media/external/file/1000213214`

#### 根因分析
Android 通过 `ACTION_VIEW` intent 打开 .recp 文件时，intent 中携带的是 content URI（如 `content://media/external/file/...`）。Flutter 引擎的深度链接机制会自动将 intent data URI 作为路由发送给 go_router。由于 go_router 没有匹配 `content://` 路由的规则，导致报错。

#### 修复方案
在 go_router 中添加 `errorBuilder`，当遇到无法匹配的路由（包括 content URI）时显示首页而不是错误页面。`receive_sharing_intent` 会单独处理文件并通过 `router.push('/playback-external/$sessionId')` 导航到正确的回放页面。

### 2026-05-08（彻底修复 .zip 后缀问题）

#### 问题
- 之前的修复（简单的 `File.rename()`）在实际设备上不起作用，文件仍然被保存为 `xxx.recp.zip`

#### 深入分析
- 排查 `file_picker` 11.0.2 的 Android 源码（`FileUtils.kt`）
- **根因链**：
  1. `FilePicker.saveFile()` → `ACTION_CREATE_DOCUMENT` intent
  2. `getMimeTypeForBytes()` 使用 Apache Tika 检测 bytes 的 MIME 类型
  3. `.recp` 文件本质是 ZIP → Tika 检测为 `application/zip`
  4. Android SAF 根据 MIME 类型自动追加 `.zip` 扩展名
  5. `FilePickerDelegate.saveFile()` 返回 `newUri.path`（content URI 路径，非文件系统路径）

- **content URI 路径示例**：`/document/primary:Download/test.recp.zip`（非 `/storage/emulated/0/Download/test.recp.zip`）

#### 修复方案（方案 B 改进版：多策略重命名）
- 在 `_fixZipExtension()` 中实现多策略重命名：
  1. **策略 1**：直接在返回路径上执行 `File.rename()`（适用于返回文件系统路径的设备）
  2. **策略 2**：将 content URI 路径转换为文件系统路径后执行 `File.rename()`（通过 `_contentUriPathToFilePath()` 解析 `/document/primary:...` 格式）
  3. **降级方案**：`rename()` 失败时使用 `copy() + delete()`
- 新增 `_tryRenameWithFallback()`：封装 rename + copy+delete 降级逻辑
- 新增 `_contentUriPathToFilePath()`：将 content URI 路径转换为文件系统路径
- 添加完整的 `debugPrint` 日志链，覆盖每一步操作

  - 新增 `_fixZipExtension()` 方法：多策略重命名
  - 新增 `_tryRenameWithFallback()` 方法：rename + copy+delete 降级

### 2026-05-08（修复打开外部文件时报 database_closed 错误）

#### 问题描述
用户反馈：某些时候打开外部 .recp 文件时会报错：
```
Exception: Failed to load session audio: DatabaseException(error database_closed)
```

#### 问题分析

通过代码分析，发现了以下可能的问题源：

1. **数据库被意外关闭（最可能）**：
   - `ExternalSessionService.openExternalFile()` 在处理已存在的外部会话时会调用 `cleanupExternalSession()`
   - `cleanupExternalSession()` 会关闭 `DatabaseHelper` 中的数据库
   - 但 `StorageService` 中的 `SessionDatabase` 缓存可能仍然持有已关闭的 `Database` 引用
   - 当 `AudioPlaybackService.loadSession()` 尝试访问已关闭的数据库时，就会抛出 `DatabaseException(error database_closed)`

2. **数据库缓存不一致**：
   - `DatabaseHelper` 和 `StorageService` 各自维护数据库缓存
   - 当 `DatabaseHelper` 中的数据库被关闭时，`StorageService` 中的缓存不会更新
   - 导致 `SessionDatabase` 持有已关闭的 `Database` 引用

3. **异步操作的时序问题**：
   - `openExternalFile()` 中的多个异步操作可能以不同的顺序执行
   - 导致数据库在需要时被关闭

4. **SessionDatabase 持有的底层 Database 被关闭**：
   - `SessionDatabase` 没有检查 `Database` 是否仍然打开
   - 直接使用已关闭的 `Database` 会导致错误

5. **外部会话清理时机问题**：
   - `cleanupExternalSession()` 可能在错误的时机被调用
   - 导致数据库在使用期间被关闭

#### 修复方案

**核心思路**：在 `SessionDatabase` 中添加数据库状态检查和自动重开机制，确保在使用数据库前数据库始终是打开的。

**1. 修改 `lib/core/database/session_database.dart`：**

- 添加 `_dbHelper` 字段，保存 `DatabaseHelper` 实例
- 将 `_db` 字段从 `final` 改为可变，以便在数据库关闭时重新赋值
- 新增 `_ensureDatabaseOpen()` 方法：
  - 检查 `_db.isOpen`，如果数据库已关闭则重新打开
  - 使用 `_dbHelper.openSessionDatabase(_sessionId)` 重新打开数据库
  - 添加调试日志记录数据库重新打开的过程
- 在 `getSessionInfo()` 方法开头调用 `await _ensureDatabaseOpen()`
- 在 `getAudioChunks()` 方法开头调用 `await _ensureDatabaseOpen()`
- 修改构造函数，接受 `DatabaseHelper` 参数

**2. 添加调试日志（所有修改的文件）：**

在以下文件中添加了详细的调试日志，帮助追踪数据库的打开、关闭和使用状态：

- `lib/core/database/database_helper.dart`：
  - `openSessionDatabase()`: 记录数据库打开和缓存状态
  - `closeSessionDatabase()`: 记录数据库关闭过程

- `lib/core/database/session_database.dart`：

#### 技术要点

### 2026-05-08（修复系统返回键无法拦截的问题）

#### 问题描述
用户反馈：点击左上角返回按钮能正确弹出保存确认对话框，但使用系统返回键（物理返回键或导航栏返回键）不会触发保存确认对话框。

#### 问题诊断
通过查看 `output.log` 发现关键日志：
```
W/WindowOnBackDispatcher: OnBackInvokedCallback is not enabled for the application.
W/WindowOnBackDispatcher: Set 'android:enableOnBackInvokedCallback="true"' in the application manifest.
```

根本原因分析：
1. **go_router 13+ 与 PopScope 不兼容**：go_router 13.x 版本与 Flutter 的 PopScope 组件存在兼容性问题，导致系统返回键无法被正确拦截
2. **PopScope vs WillPopScope**：go_router 12.x 版本对 WillPopScope 的支持更稳定
3. **Android 13+ 返回键机制**：Android 13 引入了 `OnBackInvokedCallback` 机制，需要在 `AndroidManifest.xml` 中正确配置

#### 修复内容

**1. 降级 go_router 版本**

修改 `pubspec.yaml`：
```yaml
# 修改前
go_router: ^13.1.0

# 修改后
go_router: ^12.1.3
```

**2. 将 PopScope 改为 WillPopScope**

修改 `lib/pages/playback/playback_page.dart`：
```dart
// 修改前（PopScope）
if (widget.isExternal) {
  return PopScope(
    canPop: !(externalState?.isDirty ?? false),
    onPopInvokedWithResult: (didPop, result) {
      if (!didPop) {
        _showSaveConfirmDialog();
      } else {
        _cleanupExternalSession();
      }
    },
    child: scaffold,
  );
}

// 修改后（WillPopScope）
if (widget.isExternal) {
  return WillPopScope(
    onWillPop: () async {
      if (externalState?.isDirty ?? false) {
        _showSaveConfirmDialog();
        return false; // 不退出，等待用户选择
      } else {
        await _cleanupExternalSession();
        return true;
      }
    },
    child: scaffold,
  );
}
```

**3. 修改 AndroidManifest.xml 配置**

修改 `android/app/src/main/AndroidManifest.xml`：
```xml
<!-- 修改前 -->
android:enableOnBackInvokedCallback="false"

<!-- 修改后 -->
android:enableOnBackInvokedCallback="true"
```

注意：虽然最终使用了 `WillPopScope`（go_router 12.x），但仍保留 `enableOnBackInvokedCallback="true"` 配置以保持与 Android 13+ 新返回键机制的兼容性。

#### 技术要点

- **go_router 版本兼容性**：go_router 13+ 版本引入了新的路由机制，与 `PopScope` 存在冲突。降级到 12.1.3 是稳定的解决方案
- **WillPopScope 使用模式**：`onWillPop` 回调返回 `Future<bool>`，返回 `false` 阻止返回，返回 `true` 允许返回
- **Android 13+ 返回键机制**：Android 13 引入了 `OnBackInvokedCallback` 机制，需要在 `AndroidManifest.xml` 中正确配置
- **必须重新安装应用**：修改 `AndroidManifest.xml` 后，必须卸载旧应用后重新安装，仅热重启无效

#### 验证

- `flutter pub get` 成功
- `flutter analyze` 通过，0 issues
- 重新安装应用后，系统返回键能正确触发保存确认对话框
